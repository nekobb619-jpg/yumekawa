# AGENTS.md — 「おしえて！ラボ」開発の指示書（Antigravity CLI用）

このファイルは Antigravity CLI（Gemini CLIの後継）が起動時に自動で読み込む、
このプロジェクト専用の指示書です。
`CLAUDE.md`（Claude Code / Claude用）と **内容は基本的に同じ** にしてあります。
Claude と Antigravity のどちらが作業しても、同じルール・同じ設計方針に沿うようにするためです。

**このファイルを直接いじる場合は、`CLAUDE.md` 側にも同じ変更を反映してください（逆も同様）。**
2つのAIで並行開発するときの一番の事故は「片方だけ知っているルール」が生まれることです。

## プロジェクト概要
小学4年生向けの総合学習アプリ「おしえて！ラボ」。既存の本番アプリ
「ゆめかわ★学びアドベンチャー LAB Pro」（GitHub Pages + Google Apps Script/Sheetsバックエンド、
リポジトリ `nekobb619-jpg/yumekawa`）の上に、AIの友達キャラと会話しながら
体験・実験・講義で学び、「友達に教える」→「問題をつくる」を通して
**問題のねらい（意図）を見ぬく力** を育てる機能を積み上げている。
問題は **4年生の単元を優先**、途中で **3年生までの前提スキルを診断して穴うめ**（スパイラル方式）。

## 技術スタック
- フロントは素のHTML/CSS/JS（`index.html` 1ファイル中心。ビルドツールなし、そのままGitHub Pagesで配信）
- ラボ（体験・実験パート）は `lab/教科/xxx.html` として個別ファイル、GitHub Pagesの絶対URLで参照
- データ保存：localStorage / IndexedDB（ログイン不要）／進行や記録はGoogle Apps Script経由でスプレッドシートにも保存
- AI（ニコとの会話）：将来的にClaude APIを **サーバー側プロキシ経由** で呼ぶ設計（現状は会話部分はモック／固定文言が中心）

## 現在の実装アーキテクチャ（重要・必読）
このプロジェクトは当初の設計書（`docs/requirements.md` 等）から、実装を進める中で
以下の運用パターンが確立している。**新しく作業するときはこのパターンを踏襲すること。**

### 1. content.js「1ファイル方式」
- `content.js` 1ファイルに全ステージ一覧（`STAGES`配列）と全問題データ（`QUIZZES`オブジェクト）をまとめている。
- `index.html` の末尾で読み込まれ、`window.injectContentStages()` が
  `window.globalStageMaster` / `window.availableSubjects` にステージを合流させる（`renderSubjectsNav` /
  `renderStageMaps` / `launchQuest` / `launchWeakAttackLab` をモンキーパッチして統合）。
- **新しいステージを増やすときは、スプレッドシートに行を足す必要はなく、`content.js` の
  `STAGES` に1エントリ、`QUIZZES` に対応する問題配列を1つ足すだけでよい。**
- ステージIDの命名規則：`"教科/カテゴリ/id名"`（例：`"社会/ごみ/gomi01"`）。`subject` と `category` は
  実際のGASスプレッドシートのタブ名・見出しと同じ日本語文字列にすること（表記ゆれがあると別タブに分かれてしまう）。
- 各ステージのフィールド：`subject`, `category`, `id`, `name`, `reward`, `showCount`
  （1回の挑戦で出す問題数。全問題数より少なくすると毎回ランダムに変わる）, `video_url`, `lab_url`
  （`https://`始まりなら別タブで開く＝カメラ/センサー等を使うラボ向け、`./`始まりならインライン埋め込み）。
- 期間限定公開したいときは `release_from` / `release_until`（`"YYYY-MM-DD"`）をステージに足すだけでよい
  （GAS側は一切さわらない。`isStageActive()` が日付を見て自動で出し入れする）。

### 2. 問題データの書式
`QUIZZES["教科/カテゴリ/id"]` は配列で、各要素（1問）は主に次のいずれかの形：
- 選択式：`{ q, a:[選択肢4つ], c:正解のインデックス(0〜3), hint, job_title, job_desc, speech_text? }`
- 記述式：`{ q, type:"text_input", correct_answers:[正解の表記ゆれを複数], hint, rescue_hint, job_title, job_desc }`
- キャンバス図解つき：上記に `canvas_code`（`ctx`と`canvas`を使う1行のJS文字列。図形やフローチャートを描く）を追加
- 会話シナリオつき（「ねらいを見ぬこう」「ニコに教えてあげよう」等）：`scenario:[{name, icon, msg}, ...]` を追加
- 分岐図解（テンプレート穴うめ式）：`{ q, type:"branch_diagram", diagram:{...}, hint, job_title, job_desc }`。
  `a`/`c` は使わない（正解情報は `diagram.blanks[].correct` 側にある）。詳細は下記「5.5 分岐図解トレーニングツール」参照。

**各ユニットには基本的に「【問題のねらいを見ぬこう】」という会話形式の1問を必ず含める**（このアプリの目玉機能）。

### 3. ラボ（体験・実験）の「Lab & Maker」3画面パターン
新しい単元でインタラクティブなラボを作るときは、①実験（触って発見）→②ひらめき（気づきの確認）→
③問題メーカー（通常問題／ワナ問題タブ）の3画面構成にする。
**過去に作ったラボと似たインタラクション手法（同じ操作方法）を使い回さない** こと。
（面積＝タイル敷き詰め、四角形＝頂点ドラッグ、三角形＝頂点ドラッグ＋分類、仲間さがし＝ドラッグ&ドロップ分類、
角度＝線の回転、など単元ごとに手を動かす体験そのものを変えている。）

### 4. 問題管理台帳の同期
`問題管理台帳.html` の `LEDGER` 配列は `content.js` の `STAGES`/`QUIZZES` の内容と
**常に一致させる**（id、問題数、選択式/記述式の内訳、特徴タグ）。ステージを1つ追加・変更したら、
台帳側も同じタイミングで更新すること。

### 5.5 分岐図解トレーニングツール（`type:"branch_diagram"`、探究/分岐図解/…）
子どもでも「考えを図にする」練習ができる汎用ツール。テンプレートの穴うめ式（自由描画ではない）。
`index.html` 側に描画エンジンがあり（`window.renderBranchDiagramQuestion` / `window.redrawBranchDiagram` /
`window.handleBranchBlankTap`）、`content.js` 側はデータ（`diagram` オブジェクト）を渡すだけでよい。

- テンプレートは2種類：
  - `template:"if_then"`（もし〜なら フローチャート）：ノードidは固定で `start`/`cond`/`yes`/`no` の4つ。
    `yes`・`no`のどちらか（または両方）を `kind:"blank"` にして `blankId` を指定すると、そこが穴うめになる。
  - `template:"mindmap"`（中心テーマから広がる図）：ノードidは `center` が必須、それ以外は
    `slot:"top"|"left"|"right"|"bottom"` のいずれかを指定（最大4方向）。こちらも `kind:"blank"` で穴に。
- `diagram.blanks` は `[{ id, promptLabel, options:[選択肢4つ推奨], correct:インデックス }, ...]`。
  ノード側の `blankId` と `blanks[].id` が対応する。
- ラベルの改行は `\n`（キャンバス内で複数行表示される）。長すぎる文言は箱からはみ出るので、
  1行10〜13文字程度・最大3行くらいを目安にする。
- 正解タップで箱が緑色に変わり、全部の穴が埋まると「ぜんぶできた！こたえあわせ」ボタンが有効になる
  （このボタンを押した時点で通常の正誤判定と同じ報酬ロジックに合流する）。ミスタップは
  既存の苦手リスト・チャット復習機能ともちゃんと連動する（`triggerWrongAnswer`をそのまま再利用しているため）。
- 新しい教科「探究」を追加した。`window.subjectDictionary`（index.html）にラベル登録ずみ。

### 5.6 まちがえた問題のチャット復習（`window.startReviewChat` など）
1つの問題セットの最後、まちがえた問題が1問でもあれば、いつもの「クリア！」表示の前に
ニコが吹き出しで一問ずつ「もう一回いっしょに見てみよう」→解説（`job_desc`/`hint`を再利用）→
「わかった！つぎへ」で次のまちがいへ、という復習チャットが自動で挟まる（サーバーAPI不要、
あらかじめ用意した文章のみで動く「なりきりチャット」）。
`window.sessionMissedQuestions` に1問ごとの情報を貯めておき（`launchQuest`のたびにリセット）、
`showQuestionStep`が最後の問題を終えたタイミングで `startReviewChat` に分岐する。
既存の「苦手撃破ラボ」（あとで復習）とは独立した仕組みで、両方が共存する。
新しい問題タイプ（`branch_diagram`など）を追加するときも、`triggerWrongAnswer`をちゃんと呼んでいれば
自動でこの復習チャット・苦手リストの対象になる。

### 5.63 苦手リストの安定id化（qid）と、周回プレイでの数値の その場 再生成（2026-07-20 追加）
- `content.js` は起動時に、`QUIZZES` の全問題へ自動で `qid`（安定id）を振る
  （`job_title`＋`hint`＋`q`冒頭14文字からのハッシュ。手で編集する必要はない）。
  苦手リストは以後 `ステージid::qid` 形式で保存される（旧形式 `ステージid_q_インデックス番号` は、
  問題の並び順を変えると指し先がズレるという弱点があった）。
- 旧形式のまま残っている苦手リストは `window.migrateWeakQuestionIds()`（content.js）が自動で
  新形式へ書きかえる。**この関数は content.js 読み込み時ではなく、`index.html` の
  `loadGameLocal()` と `performLogin()` 成功時（＝実際のセーブデータが `window.saveData` に
  入ったあと）に呼ぶ**（content.js自体の実行タイミングでは、セーブデータの読み込みがまだ
  非同期で終わっていないため、そこで呼んでも何もマイグレードされない。一度この順序を間違えて
  実装し、Playwrightのタイミング検証で発覚したので、ここに明記しておく）。
- クリア済みステージを周回プレイすると、`regen:{kind:"hissan_divide"}` を持つ問題（今のところ
  `算数/わり算/hissan_amari02` の筆算系9問）は、`window.regenerateQuestion()`（index.html）が
  その場で数値をランダムに再生成する（q・hint・job_desc・選択肢・canvas描画をすべて作り直す。
  `window.currentQuestions` 側の該当1問だけを差し替え、`QUIZZES`本体のマスターデータは触らない）。
  再生成された問題には `qid` は元のまま引き継がれるので、苦手リストの「どのテンプレートか」の
  判定は数値が変わってもズレない。画面には「🔄 数字が新しくなったよ！」の小さいバッジが出る
  （`#regen-badge`）。初回プレイ（まだクリアしていないステージ）や苦手撃破ラボ中は再生成しない
  （苦手撃破ラボは「まさにその問題を克服できたか」を見るためのものなので、あえて数値を変えない）。
- 新しい単元に regen対応の問題を追加したいときは、問題オブジェクトに
  `regen:{kind:"hissan_divide"}` を足すだけでよい（今のところ generator の種類は
  `hissan_divide`（2けた÷2けたのわり算の筆算）のみ。別の種類を増やす場合は
  `window.regenHissanDivide` の隣に新しい生成関数を足し、`window.regenerateQuestion`の
  `if (original.regen.kind === "...")` 分岐を増やす）。まだ全ステージには展開していない
  （今回は算数/わり算/hissan_amari02のみに適用。他ステージへの展開は今後の作業）。

### 5.65 苦手撃破ラボは複数ステージにまたがるにがて問題もまとめて処理する（2026-07-19 修正）
以前の実装は `window.saveData.weakQuestions[0]`（配列の先頭）の教科だけを見て、その1ステージ分の
バンドルしか読み込んでいなかった。そのため、いろんな単元でにがて問題がたまっている状態でラボに挑戦しても、
「先頭の1ステージ分＋足りない分はダミー問題で3問に水増し」だけが出題され、**他のステージ由来のにがては
何度ラボをやっても一生消化されない**というバグがあった（「苦手ラボが3問消化したのに減らない」という
報告はこれが原因）。
修正後（`content.js`の`launchWeakAttackLab`パッチ）は、`weakQuestions`の全要素をそれぞれ
新形式（`ステージid::qid`）または旧形式（`ステージid_q_インデックス番号`）として解釈し、
`window.CONTENT.quizzes`から解決できるものはすべて対象に含める（複数ステージが混ざっていてOK）。
1件も解決できない場合のみ、旧来の単一ステージ動的読み込み（`.js`ファイルをfetchするGAS時代の方式）
にフォールバックする。（qidそのものについては上の「5.63」参照。）
ダミー問題での水増し（3問未満のときの穴うめ）は、引き続き最初の1件が属するステージのバンドルからのみ選ぶ
（複数ステージから均等に集める、まではやっていない）。

### 5.7 苦手問題にアップする基準（あえて変更していない）
「1問でもまちがえたら即座に苦手リストへ追加」という現状の基準は、2026-07-19の相談で
「取りこぼしよりは多少過剰でも安全側」という判断で維持することにした。将来ここを変える場合は
`triggerWrongAnswer`（index.html）内の該当ロジックを見ること。

### 5.8 検証してから納品する
- `content.js` を編集したら、Node.jsで `window`/`document`/`alert` をモックして `require()` し、
  ステージ数・問題数・選択肢が4つあるか・正解インデックスが範囲内か・`canvas_code` が構文エラーなく
  `new Function('ctx','canvas', code)` で読み込めるか、を機械的に検証してから納品する。
- できれば実ブラウザ（Playwrightヘッドレス等）で `index.html` を開き、新ステージがメニューに出るか、
  コンソールエラーが出ていないかも確認する（フォント/CDN読み込み失敗やpermissions-policy警告など
  オフライン環境特有の無害なエラーは無視してよい）。選択肢はランダムにシャッフルされて表示されるので、
  Playwrightで自動クリックする場合は表示順ではなく「正解の文字列」で要素を探すこと。

## 漢字検定10級 対策ロードマップ（重要・必読）

2. だい2回：学 気 九 休 玉 金 空 月 犬 見
3. だい3回：口 校 左 三 山 子 四 糸 字 耳
4. だい4回：七 車 手 十 出 女 小 上 森 人
5. だい5回：水 正 生 青 夕 石 赤 千 川 先
6. だい6回：早 草 足 村 大 男 竹 中 虫 町
7. だい7回：天 田 土 二 日 入 年 白 八 百
8. だい8回：文 木 本 名 目 立 力 林 六 五

### 1回あたりの出題フォーマット（だい1回で確定した形。以降もこの形をベースにする）
- 対象10字につき、合計 14〜16問 くらいを目安に、`showCount` はその6〜7割程度（毎回ランダムに変わる）。
- 内訳の目安：①読み 4問／②書取（ひらがな→かん字）3問／③書き順（筆順）4問／④画数 2問／
  ⑤【問題のねらいを見ぬこう】1問／⑥さいごの text_input 復習問題 1問。
- **③書き順の比重を増やす**（画数だけでなく、実際の書く順番を問う問題を多めに）。

### 漢検コンテンツ特有の注意点（絶対）
- **問題文（`q`）にふりがな（読みがな）を書かない。** 読み方そのものを問う問題なので、
  ふりがなを書くと答えが漏れてしまう（例：「雨（あめ）が ふる」の「雨」の読み方は？→アウト）。
  `job_desc` や `hint`（正解後や間違えた後に見せる解説）にはふりがなを使ってよい。
- **画数・書き順は、出題前に必ずWeb検索でファクトチェックする。** 複数の専門サイト
  （kakijun.jp、kanji.jitenon.jp など）を見て、内容が一致した確実なものだけ出題する。
  サイトによって説明が食いちがう場合（例：「王」の1画目は縦が先か横が先か情報源で割れていた）は、
  無理に出題せず見送ること。誤った書き順を教えてしまうリスクの方が、問題数が1つ減るリスクより大きい。
- 画数だけでなく、実際の「書く順番」を問う問題（例：「右」と「左」は1画目が違う、
  「火」は左右の点→人の形の順、など）を積極的に入れる。

**2〜3日に1回くらいのペースで次の回を1つ作る**のを目安にする（毎日である必要はない）。
新しい回を作ったら、`content.js` の `STAGES`/`QUIZZES` と `問題管理台帳.html` の
`LEDGER` を両方更新し、Node/Playwrightで検証してから納品する。

## ディレクトリ構成（実際の状態）
```
（リポジトリ直下）
  index.html         … アプリ本体（1ファイル、Vite等のビルド無し）
  content.js          … 全ステージ・全問題のバンドル（上記参照）
  images/              … 画像アセット
  lab/教科/xxx.html    … 各単元の体験・実験ラボ（個別ファイル）
  算数/ 国語/ 理科/ 社会/ 外国語/ 探究/ … 過去（content.js方式より前）の個別 questions.js が残っている場合がある。
                          特に指示がない限り、削除や整理はしない（現状維持）。
  docs/                … 参照ドキュメント
```
※ サーバー（GAS = Google Apps Script）は別管理。`コード.gs` はこのリポジトリには含まれない場合がある。
　GAS側の仕様を変更する提案・実装は、必ずユーザーに確認してから行うこと（現状はGASを一切さわらない方針）。

## アプリ内の文言ルール（重要）
- すべて **小学4年生に話しかける言葉**。むずかしい漢字・カタカナ語は避ける。
- **ふりがな前提**（漢字にはルビを出せるUIにする。問題文中では「かん字（ふりがな）」のように
  読みを添える書き方が定着している）。
- 主要文字は大きめ、ボタンは押しやすく。
- 画面の文言・キャラのセリフは JSON / 設定にまとめ、あとで直しやすく。

## AI・セキュリティ（絶対）
- **Claude/Gemini等のAPIキーをフロント（ブラウザ）に置かない**。必ずサーバー側プロキシに保管する。
- 会話履歴は必要最小限を渡す。個人情報（名前・住所・学校名など）は聞かない・保存しない。
- キャラのふるまい・安全ルールはシステムプロンプトで固定する。
- こわい／きずつく／大人向けの話題は出さない。話がそれたらやさしく学習に戻す。

## キャラ設定
- **ニコ**（ユニコーン・メイン）：明るくてまっすぐ、楽しいけどまじめ。上から教えず横に並ぶ相棒。
  4ステップの進行役。問題を出してもらって「わざとまちがえる」係。ほめる・引き出す。
- **クロぽん**（パステルゴス風・サブ）：ごほうび時に登場。たまにロブロックスやマリオの話でひと息。学習にやさしく戻す。
- ※クロミ等の既存キャラは真似しない（オリジナルで作る）。

## アクセシビリティ
- よみあげ（Web Speech API）、文字サイズ変更、ふりがな表示、色だけに頼らない表現。

## Claude Code との併用について
- このリポジトリは Claude（Claude Code / Cowork）と Antigravity CLI の両方で並行して開発される想定です。
- **同時に同じファイルを2つのAIで編集しない**こと。片方が作業→コミット→もう片方が
  `git pull` してから作業、の順番を守る。
- 新しい設計判断（アーキテクチャ変更、命名規則の変更など）をしたら、このファイルと `CLAUDE.md`
  の両方に反映して、次にどちらのAIが読んでも同じ前提で作業できるようにすること。

## 補足：Gemini CLIからの移行について
- 2026年6月18日にGemini CLI（個人向け）は終了し、Antigravity CLIに一本化された。
- Antigravity CLIのインストールは公式手順（`curl -fsSL https://antigravity.google/cli/install.sh | bash`
  のような形。導入前に必ず最新の公式ドキュメントで確認すること）に従う。
- 個人のGoogleアカウント（無料枠／Pro／Ultra）でログインして利用できる。
- 旧 `GEMINI.md` は使われなくなったため、このリポジトリでは本ファイル（`AGENTS.md`）に一本化する。
